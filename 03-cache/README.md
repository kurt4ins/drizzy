# Результаты сравнения стратегий кэширования

## Конфигурация теста

| Параметр                    | Значение      |
| --------------------------- | ------------- |
| Ключей в БД                 | 1 000         |
| Длительность одного прогона | 10 с          |
| Конкурентных воркеров       | 50            |
| Интервал сброса Write-Back  | 3 с           |
| База данных                 | PostgreSQL 17 |
| Кэш                         | Redis 7       |

| Стратегия     | Нагрузка            | RPS     | Lat (ms) | HitRate% | DB Reads | DB Writes        |

| Lazy          | balanced (50/50)    | 12 765  | 3.841    | 49.1%    | 32 420   | 64 005           |

| Write-Back    | write-heavy (20/80) | 69 248  | 0.716    | 99.9%    | 202      | 4 000 (4 сброса) |

## Выводы

### Lazy Loading / Cache-Aside

**Для чтения** — посредственный результат. Hit rate составил лишь 78.5% при нагрузке 80/20, а при write-heavy упал до 19.4%. Причина: каждая запись инвалидирует запись в кэше, поэтому при высокой доле записей кэш постоянно «холодный» и почти каждое чтение уходит в БД. Это приводит к максимальному числу обращений к БД среди всех трёх стратегий: 43 039 DB reads при read-heavy и 86 196 DB writes при write-heavy.

**Для записи** — запись всегда идёт напрямую в БД, кэш только инвалидируется. Это делает Lazy самой простой реализацией, но при write-heavy нагрузке задержка вырастает до 4.6 мс — хуже, чем у конкурентов.

**Вывод:** Lazy подходит для сценариев с редкими записями и данными, которые читаются многократно после записи. При частых изменениях данных кэш практически бесполезен.

---

### Write-Through

**Для чтения** — отличный hit rate (99.1–99.7%) при всех нагрузках. Поскольку каждая запись одновременно попадает и в кэш, и в БД, данные в кэше всегда актуальны. Число DB reads минимально (189–843 за 10 секунд) — практически только первоначальное прогревание.

**Для записи** — каждая операция Set блокируется до завершения двух последовательных операций: запись в Redis + запись в PostgreSQL. Это объясняет задержку 4.5 мс при write-heavy, сравнимую с Lazy. DB writes высокие (66 460–88 565) — каждое обновление дублируется в обе системы синхронно.

**Для смешанной нагрузки** — наилучший баланс между согласованностью и производительностью чтения. При balanced (50/50) throughput 17 385 RPS при hit rate 99.4% — значительно лучше Lazy (12 765 RPS, 49.1%).

**Вывод:** Write-Through — оптимальная стратегия для смешанной нагрузки, когда важна консистентность данных в кэше. Недостаток — двойная нагрузка на БД при записи.

---

### Write-Back

**Для чтения** — лучший результат. При read-heavy throughput составил 249 423 RPS — в 10× быстрее Lazy и в 7.5× быстрее Write-Through. Средняя задержка 0.199 мс. Hit rate 100%: после первого обращения данные остаются в кэше без TTL.

**Для записи** — ключевое преимущество. Операция Set пишет только в Redis и dirty-буфер (оба in-memory), не затрагивая БД. Поэтому запись занимает доли миллисекунды. Реальных DB writes всего 4 000 за все три прогона — ровно по 4 сброса × ~1 000 уникальных ключей. Независимо от того, сколько раз один ключ был перезаписан за интервал, в БД уходит только последнее значение.

**Накопление и сброс** — за 10 секунд произошло 4 периодических сброса (интервал 3 с + финальный flush при Close). Каждый раз сбрасывалось ~1 000 «грязных» ключей. Это демонстрирует batching эффект: при высоком write volume количество DB writes не растёт линейно с числом запросов.

**Риски Write-Back:** данные, находящиеся в dirty-буфере и не сброшенные в БД, будут потеряны при аварийном завершении процесса. Стратегия неприменима там, где требуется гарантия durability на каждую операцию.

**Вывод:** Write-Back обеспечивает наивысший throughput и наименьшую задержку среди всех стратегий, особенно при write-heavy и read-heavy нагрузке. Подходит для систем с высоким объёмом записей, где допустима временная рассогласованность между кэшем и БД.

---

## Итоговое сравнение

### Лучшая стратегия для чтения

**Write-Back** — 249 423 RPS, задержка 0.2 мс, hit rate 100%. Write-Through на втором месте (33 204 RPS, 99.7% hits). Lazy проигрывает из-за постоянной инвалидации кэша.

### Лучшая стратегия для записи

**Write-Back** — запись занимает < 1 мс, DB writes в 15–22× меньше, чем у конкурентов. Если durability критична — **Write-Through** как компромисс: запись чуть медленнее, но данные сразу в БД.

### Лучшая стратегия для смешанной нагрузки

**Write-Through** — стабильный hit rate ~99% при любом соотношении читателей/писателей, предсказуемые задержки, полная консистентность кэша и БД. Write-Back лучше по скорости, но требует мириться с риском потери данных при сбое.
